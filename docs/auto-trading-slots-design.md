# 自動売買システム：枠・キャパシティ・受注管理 設計（レビュー⑦）

- 作成日: 2026-07-21 ／ 更新: 2026-07-21（クライアント回答を反映・**業務ルール確定**）
- ステータス: **確定（実装可）**
- 対象: 全自動（フルオート）プランの加盟者
- 関連: [semi-auto-trading-design.md](semi-auto-trading-design.md)

---

## 1. 目的・背景

自動売買は本部が仕入れ〜販売まで代行し、加盟者は資金を預けて運用する。
インフラ上、同時運用車両は全体で **200台が上限（将来 400台まで拡張可）**。
受注総量に制限を設け、可視化・予約・順序管理を行う。枠の購入・清算連携・完了ごとの管理手数料精算も扱う。

---

## 2. 確定した業務ルール（クライアント回答 2026-07-21）

| # | ルール |
| --- | --- |
| **稼働台数** | 「仕入れ中〜販売中」を稼働中（キャパ消費）、清算済みは対象外。清算後は再び「仕入れ中」に戻せる（ループ）。ただし**予約待ちがいる場合は、その順番の列に戻る**（枠は予約者へ回る）。 |
| **受注条件** | 1台ごとに「加盟者に空き枠あり」かつ「全体200台に空きあり」かつ「運用資金の条件を満たす」で受注可。 |
| **予約順** | 申し込みが早い順（先着）。ただし**本部が手動で順番を入れ替え可能**（運転資金の都合で先送りする運用のため）。 |
| **枠の初期値** | プランのデフォルトで決まる：**エコノミー=1枠／ブロンズ・プラチナ・ゴールド=2枠**。追加枠は手動設定。追加枠に応じて受注枠も連動。**本部都合の枠操作は加盟者に可視化しない**。 |
| **運用資金と枠** | **預かり金 < 100万円 は自動売買の受注をロック**。**1枠 = 運用資金400万円**をデフォルトとし、この額は**本部が加盟者ごとに設定可能**。実際に運用できる枠数は「保有枠数」と「運用資金でまかなえる枠数」の小さい方。※バグが出やすいので**動作チェック必須**。 |
| **月額管理手数料** | 上位プラン（ブロンズ／プラチナ／ゴールド）ごとに月額を設定（**将来変更前提でプラン設定に金額を持たせる**）。1台の運用にかかった月数分を、**販売済み車両の損益に割り込ませて預かり金から清算**。自動で減る方式でも清算時減算でも可、いずれも**履歴を可視化**する。 |
| **プレ運用の枠上限** | 本部が**加盟者ごと**に有効枠数を設定できる（現在プレ運用で実効1枠運用）。 |
| **200台の適用範囲** | 自動売買のみの概念（半自動は別枠）。**上限は設定値で、200→400等に拡張可能**にする。 |
| **予算振り分け（追加要件）** | 自動売買・半自動の**両枠を保有する加盟者**は、預かり金の**振り分け（自動売買用／半自動用）を加盟者側で自由に操作できる**。 |

## 3. データモデル

### 3-1. plans への追加（プランごとの初期枠・月額管理手数料）
```
default_auto_slots      int   -- プランの初期枠数（economy=1, bronze/platinum/gold=2, semi系=0）
mgmt_fee_monthly_yen    int   -- 月額管理手数料（上位プランのみ。将来変更前提）
```

### 3-2. members への追加（枠・運用資金設定）
```
auto_slots              int   -- 保有枠数（プラン既定＋購入分＋本部調整）。最大10。
capital_per_slot_yen    int   -- 1枠あたり必要運用資金（既定 4,000,000。本部が加盟者ごとに設定可）。
```
- 自動売買のロック閾値 100万円は全体設定（system_settings）。

### 3-3. system_settings（新規・本部が調整）
```
key         text pk    -- 'auto_capacity_total'(=200,拡張可) / 'auto_min_deposit'(=1,000,000)
value_int   int
```

### 3-4. 予算の振り分け（両フロー保有者向け）
預かり金（member_ledger）を「自動売買用」「半自動用」に振り分ける。案：
```
member_budget_alloc  -- member_id, auto_allocated_yen, semi_allocated_yen（加盟者が操作）
```
- 受注/精算の残高判定は、各フローの割当額に対して行う。
- 片方フローのみの加盟者は全額そのフローに割当（振り分けUIは両フロー保有者のみ）。

### 3-5. 受注待ち（予約）: auto_reservations（新規）
```
id, member_id, requested_at, status('waiting'|'assigned'|'cancelled'), sort_order(本部の手動並替), note
```

### 3-6. 流用（新規テーブル不要）
- 枠購入請求 = `invoices.kind='slot_fee'`（消込確定で auto_slots += 購入数）。
- 月額管理手数料 = 預かり金からの清算（ledger settlement）＋履歴（既存 ledger_entries を流用）。
- 全自動案件の進捗 = 既存 `vehicle_deals` ＋ 車両進捗管理カンバン。

## 4. 主要ロジック

### 受注可否（全自動オーダー時）
```
lock if 自動売買用の預かり金 < 100万円（system: auto_min_deposit）
稼働台数(member) = member の 仕入れ中〜販売中 の全自動案件数
有効枠(member)   = min(auto_slots, floor(自動売買用の預かり金 / capital_per_slot_yen))
受注可 = 稼働台数(member) < 有効枠(member)
         and 全体稼働台数 < auto_capacity_total(=200)
不可なら → 予約(auto_reservations, waiting) に入る
```

### 清算後のループ・予約順戻し
```
案件を清算（sold）→ その加盟者の稼働台数が1減る
if 予約待ちが存在 → 空いた枠は予約列の先頭（本部が並べた順）へ割当(assigned)
else → 同じ加盟者が続けて次を「仕入れ中」で起票可能
```

### 月額管理手数料の精算
```
車両の清算時：経過月数 = 運用開始(仕入れ中)〜清算 の月数（切上/切下は要運用確認・既定は開始日基準の満了月）
手数料 = 経過月数 × プランの mgmt_fee_monthly_yen
→ 預かり金から settlement として差し引き、ledger_entries に履歴を残す（可視化）
```

## 5. 画面（案）

- **本部：自動売買キャパ・受注管理**（新規 /admin/auto-capacity）
  - 全体稼働 ◯/200台（上限は設定値）・使用率、加盟者別の 保有枠/有効枠/稼働/空き、
    受注待ちリスト（先着順＋**手動並替**）、枠割当の実行。
- **本部：加盟者個別**（/admin/members/[id] に追記）
  - 自動売買：保有枠・有効枠・稼働、1枠あたり運用資金の設定、枠購入請求（slot_fee）、月額管理手数料の履歴。
- **加盟店：自動売買の状況**（/portal/auto）
  - 保有枠・稼働・空き、受注可否、予約状況（順番）、自分の全自動案件の進捗フロー。
- **加盟店：予算振り分け**（両フロー保有者）：自動売買用／半自動用の割当を操作。

---

## 6. フェーズ分割（実装順）

1. **データモデル基盤**：plans（default_auto_slots / mgmt_fee_monthly_yen）、members（auto_slots / capital_per_slot_yen）、system_settings、既存加盟者へのプラン既定枠の反映。
2. **受注可否・運用資金制御**：<100万ロック、有効枠=min(枠, 資金/400万)、全体200台判定。**動作チェック重点**。
3. **キャパ可視化**：本部ダッシュボード（◯/上限）＋加盟店の受注可否表示。
4. **予約（受注待ち）**：auto_reservations、先着＋本部手動並替、清算後の予約順戻し。
5. **枠購入の清算連携**：slot_fee 請求 → 消込確定で枠加算。
6. **月額管理手数料の経過精算**：プラン月額 × 経過月、清算時に預かり金差引＋履歴可視化。
7. **予算振り分け**：両フロー保有者の預かり金割当（加盟者操作）＋各フローの残高判定連動。
8. **自動売買の進捗フロー表示**：加盟者ごとの全自動案件フロー（既存カンバンを個別ビュー化）。

---

## 7. 残る細部（実装時に運用確認）

- 月額管理手数料の**経過月数の丸め**（開始日基準の満了月か、暦月か）。
- 予算振り分けの粒度（案件ごとに拘束するか、フロー全体でプールするか）。既定はフロー全体プール。
- 本部の枠操作を「加盟者に可視化しない」範囲（枠数は見せるが、本部調整の履歴は見せない等）。
